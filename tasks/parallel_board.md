- 用户已否定“图片改名 + ZIP + ID 匹配”作为主流程；后续改为每条文本计划旁边上传单张/多张图片，按选择顺序执行。

- 已确认：第一阶段采用强约束模板；同时支持 `.txt` 上传和页面粘贴框。

## 工作流 B：结构化结果协议与 UI 展示

状态：已完成第一轮只读分析

Agent：Ramanujan

目标：

- 设计逐条计划结果协议，覆盖创建状态、链接、复核清单、失败原因、截图/日志证据。
- 评估如何从 Playwright 引擎传给 UI，减少日志正则解析。

建议写入范围：

- 第一轮只读分析，不改代码。
- 后续实现时尽量新增 JSONL 结果文件和解析函数。

验收：

- 给出 JSON/JSONL schema。
- 给出 UI 任务表需要新增的字段。
- 给出兼容旧日志解析的过渡方案。

第一轮结论：

- 推荐新增 `--result-jsonl <path>`，Playwright 写结构化 JSONL 事件。
- stdout 同步输出 `@@PM_RESULT@@ {...}` 哨兵日志，便于 UI 实时消费并兼容现有日志。
- UI 任务表新增创建结果、复核状态、复核链接、失败原因、保存信号。
- 旧日志只作为 fallback；弱成功且未复核通过时必须显示 `needs_review`，不能算业务成功。
- 任务最终成功应以后续结构化结果与 required review checks 为准，不只看子进程退出码。
